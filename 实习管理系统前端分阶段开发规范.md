# 实习管理系统前端分阶段开发规范

> 最后更新：2026-06-30
> 适用范围：两人协作 + AI 辅助开发
> 页面清单唯一依据：`实习管理系统页面开发总控进度表.md`

## 0. 当前阶段：第二轮联调验收

第一轮 A/B 两线已有页面产出已经完成，作为第二轮代码/原型基线；《项目分工》中的第二轮任务目前一项都未执行。现阶段所有工作按“未开始 → 验收中 → 已验收”推进，A 线任务全部由我执行。

### 0.1 分工

* A 线（我）：学生端、教师端、学院学生申请复核、学校非企业审核页面、通用消息中心和总路由协调。
* B 线（同学）：公共认证、企业端、学院企业初审、学校企业复核、辅导员端和系统管理端入口。
* A 线兜底职责是组织联调、定位问题归属和跟踪阻塞，不代表默认接管 B 线开发。

### 0.2 第二轮允许的改动

* 修复路由、入口、返回路径和按钮行为
* 对齐同一业务状态在不同角色端的文案
* 清理占位文字、明显视觉问题和控制台错误
* 整理 mock 数据，使同一链路状态结构一致
* 在不改变已确认业务规则的前提下抽取稳定复用组件

### 0.3 第二轮禁止的改动

* 未确认业务规则前自行补全审核流向
* 因局部页面需求大范围重写公共 Layout 或总路由
* 为追求组件化破坏已确认的 Figma 视觉结构
* 未沟通就修改另一条工作线的主责页面
* 把“路由能打开”直接标记为“已验收”

### 0.4 验收状态定义

| 状态 | 定义 |
|---|---|
| 第一轮基线 | 页面文件、原型或路由已经存在，但第二轮任务尚未执行 |
| 未开始 | 第二轮对应分工项尚未开始 |
| 验收中 | 正在检查入口、行为、视觉和控制台问题 |
| 已验收 | 满足本规范验收清单，且跨端链路没有阻塞问题 |
| 占位 | 仅提供路由落点，尚无独立设计或完整内容 |

### 0.5 文档同步顺序

1. 页面和路由事实记录到总控进度表。
2. 路由或跳转变化同步到页面跳转关系与路由规划。
3. 协作规则变化同步到本规范和《项目分工》。
4. `自己分到的工作.md` 仅作为 A 线个人清单，不作为全项目事实来源。

## 1. 总体原则

本项目分两个阶段推进，不同阶段目标和约束不同：

| | 阶段一：原型还原 | 阶段二：正式工程开发 |
|------|-------------|-----------------|
| 技术 | Vue 3 + Tailwind CSS + Vue Router | Vue 3 + Vue Router + Pinia + Element Plus |
| 目标 | Figma 视觉一致 | 工程规范、可维护、可复用 |
| 布局 | div 模拟侧边栏/卡片/表格 | el-menu / el-table / el-form / el-dialog |
| 数据 | 静态 mock | 结构化 mock，可替换接口 |
| 工期 | 先快速出页面 | 逐步替换组件 |

**核心约束（两个阶段通用）**：
- 以 `实习管理系统页面开发总控进度表.md` 为唯一页面清单
- 不允许 AI 自行新增页面、路由或业务逻辑
- 阶段一不着急改成 Element Plus；阶段二不破坏阶段一已确认的页面结构

## 2. 阶段一：原型还原规范

### 技术栈
- Vue 3（`<script setup>` + Composition API）
- Tailwind CSS（`@import "tailwindcss"`）
- Vue Router（基础跳转，不做守卫、不做权限）
- 静态 mock 数据

### 目标
1. Figma 页面 1:1 还原（布局、间距、颜色、字号、卡片外观）
2. 页面数量不漏（对照总控进度表逐页交付）
3. 基础跳转可点（侧边栏、按钮、面包屑）
4. 为阶段二保留底稿结构

### 允许
- Tailwind class 自由使用
- `div` 模拟表格、卡片、侧边栏
- 静态 mock 数据写在页面内或简单 `data/` 文件
- 弹窗用自定义 `div` + `v-if` 模拟
- 页面内写简单 `ref` / `reactive` 状态

### 禁止
- ❌ 引入 Element Plus 强行替换 Figma 样式
- ❌ 重构业务逻辑
- ❌ 自行新增总控表之外的页面
- ❌ 为组件化破坏 Figma 视觉还原
- ❌ 待确认交互上脑补业务规则（只能写 `TODO`）
- ❌ 写后端接口调用

---

## 3. 阶段二：正式开发规范

### 技术栈
- Vue 3 + Vue Router + Pinia + Element Plus

### 目标
1. 统一 Layout（LayoutBackend / LayoutBlank）
2. 统一 router（按角色分模块）
3. 统一菜单（el-menu + router 联动）
4. 统一 Element Plus 组件替代手写结构
5. 统一 mock 数据结构（`mock/` 集中管理）
6. 为后续接口接入预留结构

### 替换对照表

| 阶段一手写 | 阶段二替换 |
|-----------|----------|
| `<div>` 模拟表格 | `<el-table>` |
| `<div>` + `<input>` 表单 | `<el-form>` + `<el-form-item>` |
| `<div>` + `v-if` 弹窗 | `<el-dialog>` |
| 手写分页 `<div>` | `<el-pagination>` |
| 手写日期 input | `<el-date-picker>` |
| 手写状态圆角标签 | `<el-tag>` |
| 手写下拉 | `<el-select>` |
| 手写按钮 | `<el-button>` |
| 手写侧边栏 | `<el-menu>` |

## 5. 路由规范

### 角色路由前缀

| 角色端 | 前缀 | 示例 |
|--------|------|------|
| 公共区 | `/login` `/sso/*` `/notifications` | `/login` `/sso/account` |
| 学生端 | `/stu/*` | `/stu/dashboard` `/stu/internships` |
| 企业端 | `/ent/*` | `/ent/dashboard` `/ent/jobs` |
| 教师端 | `/tut/*` | `/tut/dashboard` `/tut/applications` |
| 学院管理端 | `/col/*` | `/col/dashboard` `/col/companies/review` |
| 学校管理端 | `/sch/*` | `/sch/dashboard` `/sch/internship-plans` |
| 辅导员端 | `/cou/*` | `/cou/dashboard` `/cou/students` |
| 系统管理端 | `/sys/*` | `/sys/dashboard` `/sys/users`（P2） |

### 规则
1. 所有路由必须来自总控进度表，不允许 AI 自行新增
2. 待确认页面路由标 `// TODO: 待确认路由`
3. 弹窗/抽屉不单独建路由（除非总控表明确要求独立页）
4. 阶段一路由可以简单 `createRouter` 全量注册，阶段二再拆分模块 + lazy load

## 7. 样式规范

### 阶段一
- 优先 Tailwind，允许 Figma 导出的定位/间距/颜色
- 页面级 class 较多可接受，目标是视觉还原
- 页面背景统一 `#F5F7FB`，侧边栏 `#00203D`

### 阶段二
- 逐步减少页面内重复 Tailwind
- 公共样式 → `styles/global.scss`
- Element 主题覆盖 → `styles/element-theme.scss`
- 主色 `#2563EB`（覆盖 Element 默认 `#409EFF`）
- 按钮圆角 `24px`，输入框圆角 `8px`，卡片圆角 `15px`
- 不追求像素级，但布局和主色要接近 Figma

---

## 8. 数据与接口规范

### 阶段一
- 数据可以写在页面内或 `mock/` 文件
- 表格数据、下拉选项、状态枚举尽量抽成变量

### 阶段二
- mock 数据集中放在 `mock/` 目录，按角色拆分
- 数据结构标准化（数组 + 字段统一命名）
- 为后续 `axios` 替换预留接口结构

```js
// mock/stu.js 示例
export const internshipList = [
  { id: 1, company: 'XX科技', type: '毕业实习', status: '进行中', ... },
]
```

## 9. 页面开发流程

### 阶段一每次流程

```
1. 从总控进度表选择 1-5 个页面
2. 确认页面编号、Figma Node ID、角色、路由
3. Vue + Tailwind 还原页面（参考 Figma 截图/原型）
4. 接入路由（如果路由文件还没建，先建占位）
5. 处理按钮跳转：能用路由的就写 router-link，不确定的写 TODO
6. 本地运行确认能打开
7. 更新总控进度表「开发状态」列
```

### 阶段二每次流程

```
1. 选择已完成阶段一的页面
2. 判断需要替换的 Element Plus 组件
3. 替换组件、接入 Layout
4. 整理 mock 数据到 mock/ 目录
5. 本地运行检查
6. 更新总控进度表
```

## 10. AI 开发约束

每次给 AI 派任务时，AI 必须遵守：

1. 每次只开发总控进度表中指定的 1-5 个页面
2. 开发前说明当前属于**阶段一**还是**阶段二**
3. 开发前列出将**新增/修改的文件清单**
4. 不允许自行新增页面、路由、目录、npm 依赖
5. 遇到待确认内容只能写 `// TODO: 待确认 — {原因}`，不允许脑补
6. 阶段一不允许强行改成 Element Plus
7. 阶段二不允许大范围重构无关页面
8. 完成后必须汇报：**改了哪些文件、如何访问页面、还剩哪些 TODO**

## 11. Git 协作规范

- 每次完成 1-5 个页面就提交，不要攒几十个
- commit message 格式：

```
prototype(stu): add internship list and overview pages
prototype(ent): add job management page
feat(stu): convert internship list to Element Plus
fix(router): adjust student route prefix
```

- 合并前必须本地 `npm run dev` 能跑
- 两人各自开分支，不要直接 push main

## 12. 验收规范

### 阶段一验收标准（5 条）

| # | 标准 |
|---|------|
| 1 | 页面能通过路由打开 |
| 2 | 视觉大体还原 Figma |
| 3 | 主要入口和按钮有跳转或 `TODO` 标注 |
| 4 | 页面编号、路由、Node ID 已记录在代码注释中 |
| 5 | 总控进度表「开发状态」已更新 |

### 阶段二验收标准（8 条）

| # | 标准 |
|---|------|
| 1 | 页面能通过路由打开 |
| 2 | 菜单能正确定位到该页面 |
| 3 | 使用了规定的 Element Plus 组件 |
| 4 | 主要按钮有跳转、弹窗或 `TODO` |
| 5 | 样式接近 Figma |
| 6 | 浏览器控制台无明显报错 |
| 7 | 总控进度表状态已更新 |
| 8 | **另一个人交叉检查后才能标记"已验收"** |

## 13. 禁止事项

```
❌ 自行新增总控表外页面
❌ 阶段一就用 Element Plus 破坏 Figma 还原
❌ 多个页面重复写最终版侧边栏（阶段二用 LayoutBackend 统一）
❌ 混用多个 UI 框架
❌ 弹窗强行做成独立路由页面
❌ 修改与本次任务无关的角色端
❌ 待确认项脑补业务逻辑（只能写 TODO）
❌ 一次性生成整个项目
❌ 两个人同时改同一个文件（先沟通再动手）
```
