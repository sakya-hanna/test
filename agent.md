## 文档管理规则

本项目页面相关文档按「一主两辅」管理：

* 页面定义看 `实习管理系统页面资产台账.md`

  * 负责页面编号、名称、角色归属、模块、类型、状态、优先级。
* 完成进度看 `实习管理系统原型进度表.md`

  * 负责 Figma 页面名、Node ID、是否已画、是否已审查、待补画状态。
* 点击跳转看 ` 实习管理系统页面跳转关系与路由规划.md`

  * 负责入口、按钮、目标页、弹窗/详情页、返回路径、前端路由。

修改顺序固定为：

1. 先改资产台账，确认页面编号、归属、类型、状态、优先级。
2. 再改 Figma，页面命名统一为 `{新编号}：{页面名称}`。
3. 再改进度表，补 Node ID、审查状态、完成状态。
4. 最后改跳转关系文档，补入口、按钮、返回路径和路由。

冲突处理：

* 页面定义冲突，以资产台账为准。
* 进度状态冲突，以进度表为准。
* 跳转关系冲突，以跳转关系文档为准。
* 不确定的内容标记为「待确认」，不要擅自补业务规则。



## 组件复用（最高优先级）

**每个新建页面必须优先复用已有组件，禁止从零手绘已有组件能覆盖的部分。**

### 核心复用组件

| 组件 ID | 名称 | 用途 | 使用方式 |
|---------|------|------|---------|
| `1:13` | 左侧导航栏 | 顶栏（含 LOGO、导航菜单项） | `compNode.createInstance()`，放置于 x=232, y=0 |

### 新建学生端页面标准流程

```
1. 创建 Frame 1440×1024
2. 创建侧边栏背景 Frame（232×1024, #00203D）
3. 实例化组件 1:13 → x=232, y=0（自带 LOGO + 导航菜单）
4. 覆盖：面包屑、用户名、角色标签、登出按钮
5. 填充页面特有内容（卡片/表单/表格等）
```

### 原则

- 尽可能多地复用已有组件，尽可能少地创造新组件（非必要不创造）
- 先扫描已有组件，能用则用，不重复造轮子
- 保持视觉一致性，避免同一元素在不同页面风格各异
- 如遇少量重复元素，优先用编组（Group）复制而非创建新组件

## 节点编组规范（最高优先级）

**页面内容绘制完成后，必须按功能区域将松散节点编组（`figma.group()`），方便整体移动和管理。**

### 标准分组结构

每个 Desktop Frame 内的顶层结构应为：

```
Desktop-{页面名称}
├── FRAME  侧边栏背景          # 深色背景
├── RECT   主内容区域          # 浅灰背景
├── INST   左侧导航栏          # 复用组件 1:13
├── 📁 头部标题组              # 面包屑 + 页面标题 + 副标题
├── 📁 用户信息组              # 用户名 + 角色 + 登出按钮
├── 📁 {页面特有内容组1}
├── 📁 {页面特有内容组2}
└── ...
```

### 编组原则

- **一个功能区域 = 一个 Group**，不跨区域混合
- 通用组（每个页面都有）：`头部标题组`、`用户信息组`、`操作按钮组`
- 页面特有组按实际内容命名，如 `统计卡片组`、`数据表格组`、`周报表单组`、`历史周报组`
- Group 之间互不嵌套，保持扁平
- 编组使用 `figma.group(nodes, parent)` 自动处理坐标转换

### 编组代码模式

```js
// 分类收集 → 批量编组
const groups = {};
for (const node of allLooseNodes) {
    let key = '_other';
    if (isInSet(node, ["面包屑", "页面标题", "副标题"])) key = '头部标题组';
    else if (isInSet(node, ["用户名", "角色标签", ...])) key = '用户信息组';
    // ... more classification ...
    else key = '数据表格组';
    if (!groups[key]) groups[key] = [];
    groups[key].push(node);
}
for (const [name, nodes] of Object.entries(groups)) {
    figma.group(nodes, frame).name = name;
}
```

## 导航与路由完整性格局（强制）

**每个导航菜单项必须有对应的路由页面，禁止菜单项点击无响应。**

- 侧边栏 / 顶栏 / 快捷入口中的每一个可点击项，都必须对应一个已注册的路由和可渲染的 Vue 页面（即使是占位页面）。
- 先确保「能跳转、有页面」，再逐步按 Figma 还原页面内容。
- 新建页面时同步检查：该页面的入口在哪里？入口是否已可点击？入口最终跳转目标是否一致？
- 优先创建占位页面（含 LayoutBackend + 页面标题 + 占位提示），保证导航闭环可点击验证。

### 页面入口可见性（强制）

**每个页面必须有至少一个可见的、可点击的入口，禁止仅靠手动输入 URL 才能访问。**

- 详情页 → 必须能从列表页的「查看」「详情」等按钮点进去
- 表单页 → 必须能从某个入口页的「新建」「编辑」「去处理」等按钮点进去
- 子页面 → 必须能从父页面的导航（Tab / 侧栏 / 按钮）到达
- 新建页面后，必须回到入口页确认：入口按钮是否存在、点击后能否到达新页面

> 反面例子：实习详情页创建了路由 `/stu/internships/:id`，但实习列表页的「查看」按钮没有指向该路由——详情页就成了"隐藏页面"，用户永远找不到。

### 连贯返回路径（强制）

**页面之间要能互相跳转，要能连贯返回，禁止"进去出不来"的断头路。**

- 每个非一级页面必须提供返回上一级/来源页面的按钮（「返回列表」「取消」「← 返回」等）
- 流程页面（如申请→确认→提交）必须支持前进和后退，不能卡在中间步骤
- 弹窗/抽屉关闭后必须回到触发页，不能跳转到无关页面
- 导航链路的完整闭环：`A → B → C → B → A`，每一步都可通过按钮走完（不依赖浏览器后退键）

### 检查清单

- [ ] 侧边栏每个菜单项 → 对应路由已注册 → 页面文件存在 → 点击能跳转
- [ ] 快捷入口每个按钮 → 目标路由已注册或标记 TODO
- [ ] 卡片/列表中的「查看」「去处理」等操作 → 目标路由可解析
- [ ] 每个详情/表单/子页面 → 有至少一个入口页的按钮可以到达（入口可见）
- [ ] 每个非一级页面 → 有返回按钮可回到来源页（连贯返回）
- [ ] 面包屑路径与实际页面层级一致

## Figma MCP 工作流

直接通过 `use_figma` 工具在 Figma 文件中创建和编辑页面。前置步骤：

1. 调用 `mcp__figma__use_figma` 前，先加载 `skill://figma/figma-use/SKILL.md` 资源（ReadMcpResourceTool）
2. 调用时 `skillNames` 参数填 `"resource:figma-use"`
3. 涉及组件/变量创建时，额外加载 `figma-generate-library` 和 `figma-generate-design` 资源

### 多页面工作

- **`figma.createPage()` 不会自动切换页面！** 创建新 Page 后必须立即 `await figma.setCurrentPageAsync(newPage)`，否则所有内容会画到错误的 Page 上
- 每个 `use_figma` 调用最多切换页面 1 次
- 多页面工作需拆分为并行 `use_figma` 调用（同一消息中发出多个 tool-use block）
- `use_figma` 每次启动时默认停留在第一个 Page，非首个 Page 的操作必须先切换

### 关键规则

- `Rectangle` 不能 `appendChild`，需用 `Frame` 作容器
- 颜色范围 0-1（非 0-255）
- 文本操作前必须 `await figma.loadFontAsync(...)`
- 必须 `return` 所有创建/修改的节点 ID
